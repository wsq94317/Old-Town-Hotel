# 经济系统 & 员工系统设计（Economy & Staffing）

> 状态：**草案 / 碎片设计**（由用户口述概念 + Claude 补充建议整理，2026-06-17）
> 关联：现有 `FrontDesk2D`(Reception)、`Housekeeper2D`(HSK)、`Inspector2D`(INSP)、房间状态机、客人偏好、Day-Phase、`PlayerCash`、`Modal_DayEndSummary`。
> 注意：**游戏内文本一律英文**（编辑器无中文字体）；本设计文档用中文叙述 + 英文术语，便于对照代码。
> 这是对 **ADR 0004「单工人瓶颈是故意设计」** 的有意演进——从"单人忙碌"升级为"多员工 + 经济压力"。

---

## 1. 设计目标 / Pillars

把核心可玩点从「忙碌 + 解锁能力辅助经营」**扩展**为「**用钱换人手，在『撑着』与『扩编』之间做权衡**」。

- **核心张力**：房间数 / 客流 带来工作量；员工工资是主要持续支出。**人手不够 → 服务变差 → 收入下滑**；**人手太多 → 工资压垮现金流（入不敷出）**。玩家不断在这条线上做决策。
- **有记忆点的员工**：每个员工有鲜明性格（属性 + 特质 trait），制造"便宜但爱捅娄子的 Bob""好用但天天要涨薪的 Liz"这种让玩家记得住、会吐槽的角色。
- **精确的钱**：收入与支出可被玩家心算/预估，决策有依据。数值需 data-driven、可反复调平衡。

## 2. 员工角色 / Staff Roles

| 角色 | 对应现有系统 | 职责（瓶颈点） |
|---|---|---|
| **Reception**（前台） | `FrontDesk2D` | 接待排队客人、办理 check-in。瓶颈＝办理速度/排队长度 |
| **Housekeeper (HSK)** | `Housekeeper2D` | 清扫 Dirty 房间，使其可被检查/再次出租。瓶颈＝房间周转 |
| **Inspector (INSP)** | `Inspector2D` | 检查清扫完的房间 → 转为 Ready。瓶颈＝Ready 房供给 |
| **Manager**（新增） | 新系统 | 见 §2.1。负责"减轻玩家微操 + 提升团队效率" |

### 2.1 初始阵容 & Manager 的作用

- **开局只能雇 1 Reception + 1 HSK + 1 Manager**（用户设定）。INSP 可作为后期可雇角色，或前期由 Manager/HSK 兼任（待定，见 §8）。
- **Manager 该干什么？**（用户未定，我的建议）：Manager 不直接做一线工作，而是提供**团队层面的增益 + 自动化**，让玩家从"事事手动"中解放，但占一份工资：
  - 被动光环：小幅提升所有员工工作速度 / 降低出错率 / 降低客人不耐烦速度。
  - 自动指派：自动把空闲 HSK 派到优先级最高的脏房（没有 Manager 时玩家要手动 assign）。
  - 处理投诉 / 安抚客人，减少差评带来的收入损失。
  - **建议把 Manager 设计成"中后期的喘息阀"**：人手紧张时，雇 Manager 能放大现有人手的产出，作为"扩编"之外的另一条解法。

## 3. 员工属性 / Attributes（学历、能力、特长）

用户提出每人有"学历 / 能力 / 特长"。建议落成 **少量数值 + 特质标签**（原型期别太复杂）：

**数值（0–100 或 1–5 星）**
- **Speed**（效率）：完成一项工作的快慢。
- **Quality**（质量）：出错率 / 客人满意度加成的反面。低 Quality＝爱捅娄子。
- **Stamina**（体力）：能连续工作多久才"累"（见 §5 疲劳）。

**学历 / Education**：作为"出身"，决定**起始数值范围**和**能胜任哪些角色 / 可成长上限**。例：高学历起始 Quality 高、可当 Manager；低学历便宜但上限低。

**特长 / Specialty（标签）**：1–2 个正/负特质，制造性格与取舍：
- 正面例：`Charmer`（客人满意度+）、`Fast Hands`（清扫更快）、`Tidy`（清扫质量+）、`Calm`（高压不掉速）。
- 负面例：`Clumsy`（偶尔把干净房弄脏 / 办错入住）、`Ambitious`（定期要求涨薪）、`Lazy`（体力消耗快）、`Diva`（士气低时直接罢工）。

### 3.1 有记忆点的员工 = 特质组合（用户的 Bob / Liz）

- **Bob**：`低工资` + `Clumsy`（每天捅娄子：偶发把房间弄脏 / 排错房型 / 惹客人）→ 便宜但有隐性成本，适合预算紧张但要会兜底。
- **Liz**：`高 Quality` + `Charmer`（每个客人都服务得很好，满意度高、收入好）+ `Ambitious`（天天要涨薪）→ 好用但贵且难伺候，逼玩家做"加薪还是换人"的决定。
- → 即一个 **trait 系统**：每个候选人 = 数值 + 正特质 + 负特质 + 工资，玩家在"性价比"上权衡，自然产生记忆点。

## 4. 钱 / Money Model

### 4.1 收入 Income
- **房费**：客人 check-out 时结算，金额 × 满意度系数（满意度高 → 给好评/小费/复购，收入更高）。
- **Lounge 销售**：现有 `Lounge2D` 的饮品消耗 → 营收（可作为副收入）。
- （后期可扩展：好评带来的客流增长、活动/促销等。）

### 4.2 支出 Expenses
- **每日工资**：每个员工每天结算时扣（在 `Modal_DayEndSummary` 里展示）。这是**主要持续支出**。
- **一次性雇佣费 / 签约金**：雇人时付（可选，用来增加"扩编"的前期门槛）。
- **加薪**：`Ambitious` 员工的涨薪请求；拒绝则士气下降。

### 4.3 入不敷出的张力（核心）
- 开局**强制开放整间酒店 10 个房间**，但人手只够撑一部分 → Dirty 房积压 → 可出租的 Ready 房不足 → 收入封顶。
- 玩家**一开始就能多雇人**，但工资会让现金流为负（入不敷出）→ 必须算清楚"多一个 HSK 带来的额外周转收入，能不能覆盖他的日薪"。
- **精确计算**：建议把所有数值放进 ScriptableObject 配置（房费、日薪区间、清扫耗时、满意度系数…），方便反复调平衡。早期给一组示例数值（见 §6）。

## 5. 节奏 / Progression

| 阶段 | 房间 | 人手 | 体验 |
|---|---|---|---|
| **P1 启蒙** | 少量解锁（建议 3–4 间） | 1 Reception + 1 HSK（+ Manager） 够用，各司其职 | 学会循环、攒下现金缓冲 |
| **P2 扩张压力** | 解锁更多 / 强制开 10 间 | 人手不够 | 玩家决定：**撑着**（接受排队/差评，省工资）还是**扩编**（多雇人，扛工资）|
| **P3 经营深化** | 满房 + 高客流 | 多员工 + 班次 + 特质管理 | 优化阵容、处理加薪/罢工、追求利润最大化 |

**"撑着 vs 扩编"是 P2 的核心决策**，要让两条路都可行且各有代价。

## 6. 早期平衡草案（示例数值，待调）

> 仅为"能心算"的起点，需用配置表反复迭代。

- 房费：每位客人结账 `$80 × 满意度(0.5–1.5)` ≈ $40–$120。
- 日薪：Reception $50 / HSK $45 / Manager $90（高学历/高属性更贵）。
- 清扫耗时：1 间 Dirty 房 ≈ 45s（现有 HSK 计时器量级）；1 个 HSK 一天能翻台 N 间。
- 平衡目标：**P1（4 间房）单 HSK 翻台收入 > 团队日薪**，玩家能小幅盈余；**P2（10 间房）单 HSK 翻台跟不上客流**，不扩编则收入封顶、差评累积。
- **失败线**：现金 < 0 连续若干天 → game over（或触发"贷款"机制，加利息，加深张力）。

## 7. 我的补充建议（Claude，供参考）

1. **把 `Modal_DayEndSummary` 做成每日损益表（P&L）**：收入明细 vs 工资支出 vs 净利，直观强化"经济"主题，也是玩家复盘扩编决策的地方。（现成弹窗，低成本高回报。）
2. **士气 / 忠诚度（Morale）**：每个员工一个 morale 值，影响效率与"罢工/离职"概率。加薪、休息、合理排班 ↑；拖欠、拒绝涨薪、连续高压 ↓。这把 Liz 的"天天要涨薪"落地成系统。
3. **加薪/事件弹窗**：定期触发"X 要求涨薪 $Y/天"——接受（涨支出）/ 拒绝（掉 morale，可能罢工/离职）/ 解雇（省钱但缺人）。是高记忆点的决策时刻。
4. **疲劳 / 班次（Stamina + Shift）**：员工不能 24h 满速工作，累了掉速或需休息。即使房间数不变，也需要"备份人手"覆盖班次——天然强化"人手不够"。
5. **招聘候选池（候选人卡）**：每隔一段给 2–3 张候选人卡（头像 + 属性 + 特质 + 要价），玩家挑一个雇。轻 roguelite 的抽卡式决策，契合抖音/休闲调性，也制造"这人是谁"的记忆点。
6. **"撑着"要有具体玩法**：撑着不该只是"什么都不做"。可给临时手段——玩家手动顶岗（亲自清扫/接待，慢但免费）、加班（当天员工超时但 morale/成本上升）、临时关闭部分房间（少收入但减压）。让"撑"也是主动决策。
7. **全部数值 data-driven**：用 ScriptableObject 存"员工原型 / 特质 / 经济常数"，平衡靠改数据不改代码。
8. **失败/借贷机制**：现金为负的处理（直接失败 or 贷款+利息）决定整体难度曲线，建议先做"贷款缓冲 + 连续赤字才失败"，对休闲玩家更友好。

## 8. 待定决策（含我的推荐默认值）

> ✅ **2026-06-17 已确认：采用以下全部推荐默认值。** Phase 1 实现计划见 `docs/superpowers/plans/2026-06-17-economy-phase1-payroll.md`。

1. **Manager 的具体功能** → 推荐：**效率光环 + 自动指派 HSK + 处理投诉**（中后期喘息阀）。
2. **Inspector 怎么安排** → 推荐：**保留为现有角色，作为后期可雇**；前期 Ready 转换由 HSK 兼任或自动完成，降低前期复杂度。
3. **工资结算节奏** → 推荐：**每日结算（Day-End 扣除）**，在 DayEndSummary 展示，节奏清晰。
4. **失败条件** → 推荐：**贷款缓冲 + 连续 N 天赤字才 game over**。
5. **属性粒度** → 推荐：**3 数值（Speed/Quality/Stamina）+ 学历 + 1–2 特质**，原型期够用。
6. **开局房间数** → 你说"一开始就开放整间 10 间制造人手不够"；我的微调建议：**P1 先解锁 3–4 间让玩家上手，再在 P2 解锁/强制全开**，避免开局即劝退。如果你要"开局就 10 间"的硬核感，也可保留——这点想听你定。
